Beaver's Choice Multi-Agent System
Architecture, Evaluation, and Reflection Report

1. Purpose and Evidence

This report explains the strict hexagonal architecture shown in
"Customer Inquiry Decision-2026-06-14-100949.md", reflects on the implemented
Pydantic AI system, and discusses the evaluation recorded in
"test_results.csv".

The system still uses the required five-agent workflow and the full
"quote_requests_sample.csv" evaluation. The implementation is now organized so
the domain core is independent from Pydantic AI, SQLAlchemy, SQLite, Pandas,
Logfire, environment variables, and filesystem paths.

2. Architecture and Design Decisions

The project is split into four main layers:

1. Domain core

   The domain layer contains catalog matching, inventory, quote, fulfillment,
   discount, supplier delivery, and response-safety rules. It depends only on
   domain models and port protocols. This is where deterministic business
   behavior lives and where most offline tests run.

2. Ports

   Ports are `typing.Protocol` interfaces for AI planning/responding,
   repositories, telemetry, output, clock, and ID generation. They define what
   the app needs without binding the app to Pydantic AI, SQLite, Logfire, CSV,
   UUID, or system time implementations.

3. Adapters

   Adapters implement the ports. The current production adapters are:

   - `PydanticAiAgentAdapter`
   - `SqlAlchemyPersistenceAdapter`
   - `LogfireTelemetryAdapter`
   - `CsvEvaluationOutputAdapter`
   - `UuidGeneratorAdapter`
   - `SystemClockAdapter`

   This means LangChain could replace Pydantic AI behind the AI ports, and
   PostgreSQL or another store could replace SQLite behind the repository
   ports.

4. Application layer

   `WorkflowOrchestrator` coordinates the ports and domain services. It
   contains workflow sequencing, not business rules. `EvaluationRunner`
   handles the 20-scenario run, CSV output, and reviewer-friendly validation
   summary. `AppContainer` wires the production adapters.

The five agents remain:

1. Customer Inquiry Agent
2. Decision Orchestrator Agent
3. Inventory Agent
4. Quote Agent
5. Order Fulfillment Agent

The Decision Orchestrator Agent now extracts raw item phrases, dates,
quantities, and route. It does not decide final catalog names. The
domain-level `CatalogMatchingService` normalizes raw phrases using exact
matches, aliases, and fuzzy matching above a confidence threshold. Unsupported
or low-confidence products remain in `unmatched_items`.

The worker tools expose authoritative domain-service results:

- `inventory_lookup_tool` exposes `InventoryService` output.
- `quote_calculation_tool` exposes `QuoteService` output.
- `order_fulfillment_tool` exposes `OrderFulfillmentService` output.

This keeps agent text generation separate from business decisions and database
mutation.

3. Evaluation Results

The latest full evaluation produced these reviewer-visible metrics:

- Input requests processed: 20
- Rows written to test_results.csv: 20
- Workflow-error responses: 0
- Requests producing a sequential cash-balance change: 11
- Responses containing completed order references: 12
- Responses identifying partial or unfulfilled items: 11
- Final cash balance: $45,191.90
- Final inventory value: $4,212.40

After writing "test_results.csv", the evaluation script prints a compact
validation summary with row count, cash-balance changes, completed order
references, partial/unfulfilled response count, and the first five rows of
"request_id", "cash_balance", and "response". This makes the run log easier to
cross-check against the submitted CSV.

The refactor also adds offline regression tests. These tests verify catalog
matching, discount and delivery policies, response safety, transaction
commit/rollback behavior, rejected-order non-mutation, CSV output, and a full
workflow using fake AI ports. No pytest test calls the live LLM.

4. System Strengths

Replaceability

The business core no longer imports Pydantic AI, SQLAlchemy, Pandas, Logfire,
or environment configuration. AI, persistence, telemetry, output, clock, and
ID generation are explicit adapters behind ports.

Deterministic catalog matching

Informal customer wording such as "printer paper", "washi tape", "streamers",
and "poster boards" is normalized before worker agents run. Unsupported items
such as balloons, A3 paper, and tickets are consistently excluded.

Safer transaction handling

`OrderFulfillmentService` performs order feasibility checks before writes and
uses an atomic transaction port for restocks and sales. Regression tests verify
rollback and rejected-order non-mutation.

Explainable outputs

Customer responses are generated from validated plans and authoritative domain
results. `ResponseSafetyService` blocks internal terms such as transaction IDs,
cash balances, inventory valuations, total assets, restocking costs, and
technical errors.

Reviewer compatibility

`project_starter.py` remains the Docker target and still exposes the original
helper names required by the rubric, but those helpers now delegate to the
hexagonal package.

5. Remaining Limitations

The planner still uses an LLM to extract raw product phrases, route, dates,
and quantities. Deterministic catalog matching reduces risk, but malformed LLM
extraction can still affect the workflow.

The current persistence adapter is SQLAlchemy-based. Switching from SQLite to
PostgreSQL should be straightforward through the database URL, but switching
to a NoSQL database would require a new repository adapter.

The customer-facing response is still LLM-generated. The safety layer catches
high-risk output, but more deterministic response templates could further
reduce wording variability.

6. Future Improvements

1. Add structured extraction tests for the live planner.

   Keep normal pytest offline, but add opt-in live tests guarded by an
   environment variable to evaluate planner extraction quality against known
   sample requests.

2. Add customer confirmation for partial orders.

   Before fulfilling mixed supported/unsupported orders, ask the customer to
   confirm the supported subset, revised total, and delivery date.

3. Add deterministic response templates for high-risk outcomes.

   Use templates for rejections and partial orders so those responses cannot
   accidentally sound fully fulfilled.

4. Add a PostgreSQL integration profile.

   Run the same repository contract tests against PostgreSQL to prove the
   persistence port is genuinely replaceable.

5. Add a business advisor adapter.

   Analyze aggregate transactions and recommend reorder-point or pricing
   changes without directly mutating customer transactions.

7. Conclusion

The project now keeps the five-agent Pydantic AI workflow while moving business
rules into a strict hexagonal core. This makes the system easier to test,
easier to explain, and prepared for future replacement of the AI framework,
database, telemetry, or output mechanisms.
